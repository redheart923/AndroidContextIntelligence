# Atomic Database Rebuild v0.1 Design

## Status

Approved for implementation planning on 2026-07-16.

## Goal

Make the canonical graph rebuild failure-safe. A failed Java, AIDL/Binder, inheritance, service-registration, annotation, validation, or publication step must not replace the last verified graph database or leave its published reports inconsistent with it.

## Scope

Atomic Database Rebuild v0.1 covers the canonical multi-repository rebuild generated by `install_multi_repository_source_configuration_v01.sh`.

It must stage and publish one build batch containing:

- `android_context.db`;
- `workspace/` repository, inventory, capability, and execution-plan reports;
- `raw/ctags/`;
- `raw/aidl/`;
- `raw/inheritance/`;
- `raw/service/`;
- a `build-manifest.json` identifying the build batch.

It does not implement incremental graph updates, file hashing, Graph Patch, Permission Graph, or long-term snapshot retention.

## Selected Approach

Use a per-build staging directory and treat replacement of the SQLite database as the final commit point.

```text
data/staging/<build-id>/
├── android_context.db
├── workspace/
│   └── build-manifest.json
└── raw/
    ├── ctags/
    ├── aidl/
    ├── inheritance/
    └── service/
```

All graph importers receive paths inside the staging directory. No full-rebuild parser writes to the currently published database or reports.

After all validation succeeds, publication proceeds in this order:

1. Move the currently published `data/workspace` and `data/raw` into a build-specific rollback directory.
2. Move the staged `workspace` and `raw` directories into their published locations.
3. Atomically replace `data/android_context.db` with the staged database using `os.replace()` on the same filesystem.
4. Mark publication committed.
5. Remove rollback data and the empty staging directory.

The database replacement is the commit point because existing graph consumers use `data/android_context.db` as the authoritative artifact. Reports are prepared first and restored if the database has not committed.

## Components

### `workspace/build_publish.py`

This module owns build-batch lifecycle and publication recovery. It exposes a small command-line interface used by `scripts/rebuild_all.sh`:

```text
python -m workspace.build_publish begin
python -m workspace.build_publish prepare --staging <path>
python -m workspace.build_publish publish --staging <path>
python -m workspace.build_publish fail --staging <path>
python -m workspace.build_publish recover
```

The module is responsible for:

- generating collision-resistant build IDs from UTC time, process ID, and random entropy;
- creating staging and rollback directories beneath `data/` so every rename remains on one filesystem;
- writing publication state durably before moving live report directories;
- replacing the database with `os.replace()` as the final commit point;
- restoring live reports after a pre-commit publication failure;
- recovering an interrupted publication on the next rebuild;
- cleaning completed rollback and staging directories;
- returning machine-readable paths to the shell orchestrator.

### `scripts/rebuild_all.sh`

The generated canonical rebuild remains the user-facing entry point. It gains:

```text
--keep-failed-db
```

For a normal rebuild it must:

1. Parse arguments before filesystem mutation.
2. Acquire an exclusive non-blocking lock at `data/.rebuild.lock`.
3. Recover an interrupted pre-commit publication if one exists.
4. Create a staging batch.
5. Generate the workspace plan inside the staging batch.
6. Initialize and populate the staged SQLite database.
7. Write every raw report inside the staging batch.
8. Run all staged validation queries.
9. Checkpoint the staged SQLite WAL, switch it to `journal_mode=DELETE`, and verify that no staged `-wal` or `-shm` sidecar remains.
10. Write the staged `workspace/build-manifest.json`.
11. Publish the batch.

`--discover-only` and `--plan-only` retain their existing non-database behavior and do not create a database staging batch. They continue to update published workspace planning reports because those commands explicitly request those reports without publishing a graph database. Because they mutate `data/workspace`, they acquire the same lock and recover any interrupted publication before writing reports. `--help` is the only mode that exits without acquiring the lock.

### `workspace/build-manifest.json`

Every successful batch contains:

```json
{
  "build_id": "20260716T150949Z-12345-a1b2c3d4",
  "status": "verified",
  "source_config": "/home/ts/android-context-intelligence/config/source_roots.toml",
  "started_at": "2026-07-16T15:09:49Z",
  "verified_at": "2026-07-16T15:25:45Z"
}
```

The same build ID is written into the staged database as a `GRAPH_BUILD` node before validation. After publication, `data/workspace/build-manifest.json` and the database therefore prove they belong to the same batch.

## Failure Semantics

### Parser or validation failure

The live database and published reports are untouched.

- Default behavior removes the staging batch.
- `--keep-failed-db` preserves the complete staging batch and prints its absolute path.

### Publication failure before database replacement

The publisher first moves any newly published report directories back into the staging batch, then restores the prior `workspace` and `raw` directories from rollback storage. The live database remains unchanged. This preserves the complete failed batch when `--keep-failed-db` is active. The publication journal remains until recovery has completed.

### Interruption during publication

Before moving a published directory, the publisher writes a journal containing the build ID, staging path, rollback path, and publication phase. At the start of the next rebuild, `recover` reads the build ID from the live database instead of trusting the journal phase alone. This closes the interruption window between `os.replace()` and the following journal update. Recovery then performs one of two deterministic actions:

- if the live database does not contain the journal build ID: restore the previous reports and keep the previous database;
- if the live database contains the journal build ID: finish publication cleanup because the new database is authoritative.

The journal is removed only after recovery or publication cleanup succeeds.

### No previous database

If a first build fails, `data/android_context.db` remains absent. No parser may create it before final publication.

### Concurrent rebuild

`scripts/rebuild_all.sh` uses non-blocking `flock` on `data/.rebuild.lock`. Any second rebuild, discover-only, or plan-only command exits with a clear error while another graph command holds the lock. This prevents planning output from racing with batch publication.

### SQLite WAL safety

The schema enables WAL while importers populate the staged database. A main database file must never be replaced while staged or live WAL sidecars could still contribute pages.

Before publication, the publisher must:

1. execute `PRAGMA wal_checkpoint(TRUNCATE)` against the staged database;
2. execute `PRAGMA journal_mode=DELETE` against the staged database;
3. close the connection and verify that staged `-wal` and `-shm` files are absent;
4. checkpoint the currently published database when it exists;
5. refuse publication before moving reports if the live database remains busy or live `-wal`/`-shm` files remain.

It must also verify that the staging directory build ID, the staged database `GRAPH_BUILD` ID, and `workspace/build-manifest.json` build ID are identical before moving any published artifact.

The published database therefore uses the rollback-journal mode between rebuilds. Each new staged database enables WAL again through `storage/schema.sql` while it is being populated. This permits a single-file atomic commit without attaching stale WAL pages to the replacement database.

## Compatibility

Existing consumer paths remain unchanged:

```text
data/android_context.db
data/workspace/
data/raw/
```

No symlink migration is required. Existing SQL query commands and external tooling continue to work.

The clean-install contract also remains unchanged: users need only the five root installation scripts. The multi-repository installer embeds the publisher module, tests, and updated canonical rebuild script.

## Test Strategy

Tests use temporary directories and real files; they do not require an AOSP checkout.

Unit tests for `workspace/build_publish.py` prove:

- `begin` creates all expected staging directories without touching live artifacts;
- a pre-commit failure restores old reports and preserves the old database bytes;
- successful publication replaces the database and reports with one build ID;
- first-build failure leaves no live database;
- default failure cleanup deletes staging;
- keep-failed behavior preserves staging;
- interrupted publication recovery is idempotent;
- a committed publication is completed rather than rolled back.
- staged WAL content is checkpointed before publication;
- publication is rejected without changing live artifacts when live SQLite sidecars indicate an active or uncheckpointed database.

Shell integration tests prove:

- argument parsing recognizes `--keep-failed-db`;
- `--discover-only` and `--plan-only` do not create a staged database;
- a forced importer failure leaves the previous database unchanged;
- a second concurrent rebuild is rejected by the lock;
- successful rebuild validation still includes foreign keys, AMS, PMS, and LocalServices.

The existing complete test suite must continue to pass.

## Documentation and Git Boundaries

Implementation is split into reviewable commits:

1. publisher module and unit tests;
2. canonical rebuild staging and integration tests;
3. installer embedding and clean-install verification;
4. README and installation-manifest updates.

Each commit is verified independently. Generated databases, staging batches, rollback directories, journals, and locks remain excluded from Git.

## Acceptance Criteria

The feature is complete only when all of the following are demonstrated:

- a forced rebuild failure preserves the exact checksum of the previous database;
- the failed rebuild does not leave partially published workspace or raw reports;
- default failure removes staging;
- `--keep-failed-db` retains the full failed batch;
- a successful rebuild publishes matching build IDs in the database and manifest;
- interrupted publication recovery is idempotent;
- concurrent rebuild is rejected;
- no published database is accompanied by stale `-wal` or `-shm` files;
- the complete project test suite passes;
- canonical foreign-key, AMS, PMS, and LocalServices validation passes;
- a clean installation still requires only the existing five root scripts.
